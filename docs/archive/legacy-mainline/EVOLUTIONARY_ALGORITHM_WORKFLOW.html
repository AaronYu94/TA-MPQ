Workflow Overview

TA-MPQ 进化算法工作流

这份页面聚焦于 进化算法 / surrogate / closed-loop search 这一条方法线， 用来说明当前搜索系统的结构、输入输出关系，以及下一轮应如何继续推进。

当前主线：uniform8-aware
任务：MATH-500
搜索空间：{4, 8, 16}
预算目标：≈ uniform 8-bit 27B
当前目标：TA-MPQ > uniform8 > native9B
Validated Result

当前已经验证完的结果

36% TA-MPQ quantized 27B
39% uniform 8-bit 27B
29% native 9B
当前比较稳的结论是：TA-MPQ > native 9B，但相对 uniform 8-bit 的优势还没有被稳定证明。
Current Diagnosis

现在进化算法真正卡住的地方

  • 不是 backend 执行错了，而是 16-bit 预算还没分到最值钱的 group。
  • surrogate 现在已经有用，但对 “哪些组该上 16-bit” 的排序仍然不够准。
  • 所以不能只靠“再跑更多代”，要先用 targeted ablation 纠正 value prior。
当前方法不是盲搜，而是：ablation → prior → surrogate → search → PTQ → eval → 回喂。
Algorithm Role

这个进化算法在项目里的定位

它不是直接训练大模型，也不是直接跑最终 benchmark 的主体。 它的定位是一个 budget-constrained policy search engine： 在固定预算下，为每个 layer group 选择 4 / 8 / 16-bit，生成最值得拿去做真实 PTQ 的候选策略。

模块 作用 当前实现
编码 把 candidate 表示成 group-level bit allocation per_block_component 上的 {4,8,16}
fitness 不直接看真实分数，而看 surrogate 的保守预测 conservative_prediction + budget alignment + prior alignment
evolution selection / crossover / mutation / immigrants 已经做了稳定化
closed loop 只真实验证少数有价值 candidate，再把结果回灌 按 exploitation + exploration + diversity 选点
Workflow

单独进化算法工作流

1

收集真实执行记录

从 executed manifest 读取已经真实 PTQ + benchmark 过的 policy。

  • 输入：outputs/closed_loop/math500-uniform8-aware-executed-manifest-v3-ablation16.json
  • 输出：可训练的 executed-policy dataset
2

构建 surrogate dataset

把每条 policy 展开成特征，并把目标设成当前路线真正想优化的指标。

  • 当前路线的 target 是：accuracy_advantage_over_uniform
  • 也就是直接学 “比 uniform 8-bit 强多少”
3

训练 surrogate

训练一个 small-data 友好的 ensemble surrogate，用来近似预测 candidate 的真实 benchmark 表现。

  • 当前是 XGBoost ensemble
  • 输出均值预测和不确定性
  • 搜索时用的是 prediction - penalty × uncertainty
4

注入 group value prior

不让进化算法只看 surrogate，还要给它一个“哪些组值得高精度”的先验。

  • 当前 prior 来自执行数据和 targeted ablation
  • 这一步是把 “16-bit 该给谁” 这个问题显式喂回搜索
5

运行进化搜索

在 {4,8,16} 空间里搜索 top-k candidates。

  • population：32
  • generations：15
  • 支持 adaptive mutation + immigrants + 去重
  • 目标预算：≈ uniform 8-bit 27B
6

closed-loop 选点

不是真实量化所有 candidate，而是挑少数“最有价值”的点去真跑。

  • 一部分看 exploitation：当前预测强不强
  • 一部分看 exploration：不确定性高不高
  • 一部分看 diversity：和已执行 policy 差得够不够远
7

真实 PTQ + dev benchmark

对选中的 candidate 做真实量化、真实运行、真实测分。

  • 先小样本 dev（通常 limit=25）
  • 只有新 winner 才值得再上大样本
8

回喂到下一轮

把新的 PTQ report、evaluation、ablation 掉点重新写回 manifest，再进入下一轮。

  • 这一步才是 closed-loop 的核心
  • 真实分数负责纠正 surrogate 和 value prior
Current Inputs

当前这一轮进化算法真正吃的输入

  • contract：configs/experiment_contract_27b_9b_math500.json
  • executed manifest：outputs/closed_loop/math500-uniform8-aware-executed-manifest-v3-ablation16.json

Recommended Next Step

当前更值得优先推进的工作

  1. 跑完 16 -> 8 和 8 -> 4 的 targeted ablation
  2. 把真实掉点并进新的 group value prior
  3. 重训 surrogate
  4. 重跑 uniform8-aware 的 {4,8,16} search
  5. 只把新 top-1 做 PTQ 和大样本 benchmark
当前性价比最低的做法，是继续拿旧 winner 重复跑大样本，或者仅仅把 generations 从 15 机械地抬到 30。
Next Steps

接下来的步骤

阶段 要做什么 产出 目的
1. 补齐当前 ablation 完成当前 16 -> 8 与 8 -> 4 的 g12 批次，并整理掉点结果 两个完整的 ablation profile 确认哪些 group 的高精度有真实边际收益，哪些只是占预算
2. 更新 prior 把新的 ablation 掉点并入 group value prior 新的 ablation-adjusted prior 让搜索不再主要依赖结构先验
3. 重训 surrogate 用更新后的 manifest / dataset / prior 重新训练 surrogate ensemble 新的 surrogate summary 和 model 提高对 “哪些 group 值得 16-bit” 的排序质量
4. 重跑搜索 在同样的 {4,8,16} 与 uniform8 budget 下重跑搜索 新的 top-k candidates 检验修正后的 value signal 是否会改变精度分配
5. 真实 PTQ 只对新的 top-1 做真实量化和 artifact 验证 可运行的 quantized candidate 避免在还没收敛前把预算花在太多候选上
6. 重新验证 benchmark 先跑 dev，再决定是否上新的 large-sample eval 新的 head-to-head 结果 判断是否已经有必要再次挑战 uniform8
当前建议的推进顺序是：先修正高精度分配，再重训搜索，再做新的 benchmark 验证。
Operational Commands

当前继续这条进化算法线时的常用命令

1. targeted ablation

python3 -m modal run src/ta_mpq/modal_feasibility_app.py::run_precision_ablation_sensitivity \
  --report-path outputs/feasibility/math500-qwen35-27b-vs-9b-math500-uniform8-route-v3-ablation16-candidate-01-llmcompressor-source-report.json \
  --candidate-path outputs/policies/math500-uniform8-route-v3-ablation16/candidate-01.json \
  --contract-path configs/experiment_contract_27b_9b_math500.json \
  --ranking-profile-path outputs/sensitivity/math500-27b-task-sensitivity-v2-limit-32.json \
  --ranking-field combined_sensitivity \
  --max-groups 12 \
  --floor-bit 8 \
  --allowed-bits 8,16 \
  --reference-bits 16 \
  --policy-source llmcompressor \
  --backend-variant source \
  --task-name math500 \
  --limit 25 \
  --max-new-tokens 32 \
  --calibration-limit 4 \
  --prior-weight 0.25 \
  --output-name math500-uniform8-route-v3-candidate-01-precision-ablation-16to8-g12

python3 -m modal run src/ta_mpq/modal_feasibility_app.py::build_surrogate_dataset \

System Boundary

这个搜索系统的角色边界

它不是在训练模型权重本身，而是在预算约束下搜索“哪些 layer groups 应该分配 4 / 8 / 16-bit”。

系统负责的部分

  • 生成 candidate policy
  • 用 surrogate 快速估分
  • 在预算约束下搜更好的 precision allocation
  • 只挑少量 candidate 去做昂贵的真实 PTQ

系统不直接负责的部分

  • 不直接训练 Qwen3.5-27B 权重
  • 不直接产生最终结论
  • 不替代真实 benchmark
  • 不靠单轮 search 就结束
