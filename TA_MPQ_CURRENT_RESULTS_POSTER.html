Why the route changed

The old evolutionary route mixed random mutation, crossover, surrogate ranking, and budget repair. In practice, crossover fought the exact-budget constraint and produced unstable candidates.

The new route makes the budget explicit: traverse an ordered frontier from mostly INT4 toward more INT8, using INT2 demotions only where needed to stay near the target size.

1. ProfileTask sensitivity on MMLU-coding dev prompts.
2. RankGroup value for 8-bit promotions and 2-bit demotions.
3. BuildGreedy frontier under exact INT4 footprint.
4. RefineLocal search around the best coarse region.
5. ValidateRun real quantized artifacts on held-out tasks.
Main claim today The method is now controllable and reproducible. Performance is mixed: it clearly improves over INT4 on HumanEval/EvalPlus, but it does not yet beat INT4 or INT8 on MMLU-coding.

What we can say now

  • INT8 remains close to BF16 across current coding evaluations.
  • Uniform INT4 is a strong size baseline, but loses clear accuracy on HumanEval/EvalPlus.
  • The mixed exact-budget policy improves over INT4 on HumanEval while matching the INT4 footprint.
  • MMLU-coding currently does not separate mixed from INT4; stronger selection/evaluation coverage is needed.
  • BigCodeBench is the next meaningful generative-code test once execution grading is stable.

Current interpretation

The new structured search has fixed the controllability problem from evolutionary crossover. It produces a real quantized model with a transparent bit distribution and near-zero budget slack. The empirical story is not "mixed beats everything"; it is more precise: mixed can recover meaningful INT4 degradation on code-generation transfer tasks while staying at an INT4-sized footprint.

Next steps

  • Finish BigCodeBench generation for INT8, INT4, and mixed; then run official execution grading.
  • Report end-to-end VRAM and throughput separately from raw weight footprint.
  • Use BigCodeBench results to decide whether MMLU-coding sensitivity is sufficient or needs task-specific refinement.
  • If BigCodeBench shows mixed gains, promote this route as the main TA-MPQ story; if not, tune the sensitivity profiler and candidate generator.
